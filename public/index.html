现代布局只剩两个：Flex 还是 Grid

别再用 float、inline-block、table 凑布局了。

Flex — 一维

排成一行 / 一列。元素的大小由内容决定，只关心怎么沿主轴排开。

  • 导航栏、按钮组、表单一行
  • 左侧固定宽 + 右侧自适应
  • 底部 sticky footer

Grid — 二维

同时管行和列，容器决定网格，孩子放进去就好。

  • 页面整体框架(header / aside / main)
  • 卡片墙、商品列表
  • 表格式表单(label 对齐)

/* 一句话决策 */
.row   { display: flex;        gap: 12px; }              /* 一行排开 */
.cards { display: grid;
         grid-template-columns: repeat(auto-fill,
                                       minmax(240px, 1fr));
         gap: 16px; }                                    /* 卡片墙 */
判断标准很简单：只关心「沿一个方向排」用 Flex,需要「行列同时对齐」用 Grid。
auto-fill + minmax 是卡片墙的标准范式，不需要写 media query 就能自动换行。

实战：这种排行榜怎么拆

第一眼觉得复杂，拆开看就是Flex 套 Flex。

教师 / 课程双榜，前三领奖台 + 后续列表

拆解思路

  • 外层横向 Flex：左右两块榜单 flex: 1 等分
  • 每块内部纵向 Flex：标题 + 领奖台 + 列表
  • 领奖台用横向 Flex + align-items: flex-end：第二第三矮、第一高，底部对齐
  • 下方排名用纵向 Flex + justify-content: space-between：左侧名字、右侧分数贴边

/* 外层 */
.report { display: flex; gap: 16px; }
.panel  { flex: 1; display: flex; flex-direction: column; }

/* 领奖台:三个块底部对齐,第一名居中拔高 */
.podium { display: flex; align-items: flex-end;
          justify-content: center; gap: 12px; }
.rank-1 { height: 180px; }   /* 高 */
.rank-2, .rank-3 { height: 140px; }

/* 排名行:名字左、分数右 */
.row { display: flex; justify-content: space-between;
       align-items: center; padding: 12px 0; }
复杂 UI 拿到手别慌，先画"嵌套盒子图"，每层只问一个问题:横排还是竖排。
现场可以问大家:这个 UI 的"领奖台"为什么三块底部对齐? 答案:align-items: flex-end。第一名 height 高,自然就拔起来了。 初学者最容易卡在"中间高两边矮"这种异形布局,其实就是 align-items 的事。

居中这件小事

2026 年了，居中应该是一行的事。

✓ Grid 一行流


.box {
  display: grid;
  place-items: center;
}

✓ Flex 经典


.box {
  display: flex;
  justify-content: center;
  align-items: center;
}

✓ margin auto(已知宽高)


.child {
  margin-inline: auto;  /* 水平 */
  /* 垂直需要父级是 flex/grid */
}

✗ 别再写


.child {
  position: absolute;
  top: 50%; left: 50%;
  transform: translate(-50%, -50%);
}

能用，但脱离文档流，容易把后面的事搞复杂。

place-items 同时管 align 和 justify,记一个顶两个。

选择器优先级：别再 !important

优先级是 (行内，ID, 类/属性/伪类，元素/伪元素) 四元组，逐位比较。


/* (0,0,1,0) — 一个类 */
.btn { color: red; }

/* (0,0,2,0) — 两个类 */
.card .btn { color: blue; }

/* (0,1,0,0) — 一个 ID,赢上面所有 */
#submit { color: green; }

/* 行内样式 = (1,0,0,0),还能更高 */
<a style="color: pink"></a>

/* !important 是核武，会让别人也只能 !important */

降权::where()


/* :where() 让选择器优先级永远是 0 */
:where(.reset) ul { list-style: none; }

/* 用户随便用一个类就能覆盖 */
.my-list { list-style: disc; }   /* 赢 */

写组件库或全局 reset 时，用 :where 把基础样式优先级压到 0,业务覆盖永不打架。

!important 是技术债

它能赢，但会让下一个改样式的人必须也写 !important,然后又一个，最后整个文件都是惊叹号。
正确做法：找到为啥不生效(99% 是优先级或层叠上下文),从根上改。

Chrome DevTools 的 Computed 面板能直接告诉你「被谁覆盖了」，别靠猜。

单位选哪个，clamp 真香

不同单位代表不同的「参照物」，别凭手感写。

单位 参照 什么时候用
px物理像素边框、阴影、1-2px 的细节
rem根元素 font-size字号、间距 — 全局可缩放
em当前元素 font-size按钮 padding 跟字号联动
%父元素对应方向弹性宽度、modal 蒙层
vw / vh视口大小全屏 hero、banner
dvh / svh / lvh动态/小/大视口移动端浏览器地址栏伸缩 — 用 dvh
 

/* 流体排版：小屏 16px,大屏 22px,中间线性插值 */
h1 { font-size: clamp(1.4rem, 1rem + 1.5vw, 2.2rem); }

/* 内边距也能 clamp */
.section { padding: clamp(16px, 4vw, 64px); }

/* 移动端高度别再用 100vh,会被地址栏挡 */
.hero { min-height: 100dvh; }
clamp(min, prefer, max) 几乎可以代替 80% 的 media query。

CSS 变量：主题切换的标准答案

变量 + data-* 属性，十几行实现暗色模式。

实时演示

Hello, theme.
一段示例文字，跟随主题切换颜色。 

:root {
  --bg: #fff;
  --fg: #0f172a;
  --primary: #6366f1;
}
[data-theme="dark"] {
  --bg: #0f172a;
  --fg: #e2e8f0;
  --primary: #a855f7;
}
[data-theme="candy"] {
  --bg: #fff7fb;
  --fg: #831843;
  --primary: #ec4899;
}

body {
  background: var(--bg);
  color: var(--fg);
  transition: background .2s, color .2s;
}


// 切主题就一行
document.documentElement
        .dataset.theme = 'dark';

实操要点：把所有可主题化的颜色都收拢到一组语义化变量(--bg / --fg / --primary), 业务代码只引用变量。换主题 = 改一组变量 = 全站生效。

响应式新姿势：容器查询

@media 看的是视口,@container 看的是组件自己。

老办法 @media 的痛

同一个卡片组件，放在 sidebar(窄)和 main(宽)里，需要的样式不同。
但 @media 只看视口宽度，没法知道「我现在被放在哪」。

新办法 @container

组件自己决定自己：容器宽过多少就横排，窄了就竖排。
组件可以被任意嵌套，行为永远正确。 


/* 1. 父级声明这是个查询容器 */
.card-wrap {
  container-type: inline-size;
  container-name: card;
}

/* 2. 卡片根据自己所在容器的宽度决定布局 */
.card { display: grid; gap: 12px; }

@container card (min-width: 480px) {
  .card { grid-template-columns: 160px 1fr; }   /* 宽 → 横排 */
}
现代浏览器全支持(2023+)。组件库里早该用了。

现代选择器：:has() 让父级跟着孩子变

CSS 终于有「父选择器」了。配合 :is() / :where(),选择器表达力上一个台阶。

试一下

勾选 → 整个 card 变样。没用一行 JS。


/* :has() — 父级根据子级状态变样 */
.card:has(input:checked) {
  background: linear-gradient(...);
  border-color: var(--brand-1);
}

/* :is() — 选择器分组，优先级 = 组里最高的 */
:is(h1, h2, h3) + p { margin-top: 0; }

/* :where() — 同 :is(),但优先级永远 0 */
:where(.reset *) { all: unset; }

/* 表单有错误就给整个表单提示 */
form:has(:invalid) .submit { opacity: 0.5; }
:has() 之前要靠 JS 给父级加 class,现在交给 CSS 就行，SSR 也友好。

前端三准则：组件化优先

写代码之前先问自己三件事，能省一半返工。

① 能用组件就用组件

业务里每一块"长得像"的 UI，都是组件的雏形。

  • 表单项、卡片、弹窗、空状态
  • 优先用 Element Plus 现成的
  • 抽不出来就别硬抽，三处重复才考虑

② 能复用就复用

复制粘贴是债，第三次复制时必须停下来抽。

  • UI 抽组件，逻辑抽 composable / hook
  • 常量、工具函数集中放 utils/
  • 跨项目复用 → 拆 monorepo 包

③ 低耦合 高内聚

组件像积木，不像连体婴。

  • 高内聚：一个组件只干一件事
  • 低耦合：靠 props/events 通信，不直接读父级状态
  • 不要在子组件里 $parent、不要在父组件里改子的内部 state

<!-- 反例：紧耦合，UserCard 知道太多 -->
<UserCard :user="user" @save="user.save(); $emit('refresh')" />

<!-- 正例：UserCard 只发事件，父级决定怎么处理 -->
<UserCard :user="user" @save="handleSave" />
一个组件如果要看 5 层父级才能看懂，它就该被拆。
三准则是套娃关系： 组件化是手段，复用是目的，低耦合高内聚是判断"抽得对不对"的标准。 初学者最常见的错：① 不抽组件，整个页面 800 行；② 抽了但耦合死了，组件离不开特定父级。

CSS 性能：别瞎优化

现代浏览器很聪明，大部分「优化」都是迷信。但有几条是真的。

✓ 真有用的

  • 动画只动 transform 和 opacity — 走 GPU,不触发 layout/paint
  • 大列表用 content-visibility: auto 让浏览器跳过屏外渲染
  • 关键 CSS 内联到 <head>,首屏不等外部样式表
  • 字体加 font-display: swap 避免 FOIT

✗ 别再相信的

  • 「选择器从右往左匹配，所以要短」 — 现代浏览器开销可忽略
  • 「* 通配符性能差」 — 也不会差
  • will-change 乱加 — 反而吃内存，只在动画前一刻加，完成后移除
  • 过早预渲染、过早 lazy load — 测了再说

/* 动画的好与坏 */
.bad  { animation: shake 1s; }
@keyframes shake { from { left: 0 } to { left: 10px } }   /* 触发 layout,卡 */

.good { animation: shake 1s; }
@keyframes shake { from { transform: translateX(0) } to { transform: translateX(10px) } }
不要凭感觉优化。开 Performance 面板录 5 秒，数据会告诉你真实瓶颈。

心智模型：它不是补全

Claude Code 是能读代码、能跑命令、能改文件的同事，不是按 Tab 出代码的插件。

① 意图

你要什么?
把目标说清，别让它猜。

「加个登录按钮」 → 「在 Header.vue 右上角加登录按钮，点击跳 /login,样式跟现有 nav 按钮一致」

② 上下文

它需要看哪些文件?
给路径，别让它满项目搜。

CLAUDE.md / 指定文件 / 子 agent 探索 — 都是给上下文。

③ 验证

怎么算做完?
说清验收标准，它会自己跑。

「跑 npm test 通过」「npm run dev 后页面能点击不报错」

输出质量 = f(意图清晰度 × 上下文完整度 × 验证可执行性)

这三件事少一件，产出就要打折。把这三件事说清，基本上一遍就成。
很多人吐槽 Claude Code 「不靠谱」，根因往往是上面三件事缺一两件。 比如只说「修一下这个 bug」，不给文件、不给复现步骤，出来的东西不对再正常不过。

Plan 模式 + 子 agent：复杂任务的两件套

Plan 模式

按 Shift+Tab 进入。它只读不写,先给你一份完整方案，你审完再放行。

  • 多文件改动、不熟的代码、有风险的操作 — 必走
  • plan 不对的话直接驳回，比改了再回滚便宜
  • plan 文件留档，后面能回看

Explore 子 agent

遇到大代码库别让主对话满屏读文件，派子 agent 去探。


「用 Explore agent 看下
 登录流程经过哪些文件，列出关键函数和调用链。「

子 agent 读完只回结论，主对话上下文不被污染。

Plan 模式典型流程

  1. Shift+Tab 进 plan → 描述需求
  2. 它探索代码、问关键问题、写出计划
  3. 你看 plan,有疑问追问、有偏差让它改
  4. OK 了让它退出 plan 开始实施
  5. 实施完看 diff,不对就回退
没想清楚就让它写代码，等于让你看一份没需求文档的 PR。

MCP 工具：CodeGraph 比 grep 准

MCP = Model Context Protocol,把外部工具接成 Claude Code 的「插件」。

CodeGraph 能干啥

  • 谁调用了这个方法 — callers
  • 这个方法调用了谁 — callees
  • 改这个会影响哪些地方 — impact
  • 从一个符号出发探索 — explore
  • 按名字找 class / function — search

避坑清单：把它当实习同事带

✓ 应该做

  • 永远看 diff — 提交前过一遍它改了啥
  • 复杂任务先 plan — Shift+Tab
  • 给具体路径和命令 — 别让它满项目找
  • 验证改动 — 让它跑测试 / 起 dev server
  • 常 /commit 小步快跑 — 出错好回退

✗ 别做

  • 无脑 yes — 删除/重置/force push 必须自己确认
  • 在脏工作区让它干活 — 先 commit 或 stash
  • 让它跑 prod 命令 — 部署、发布要人工把关
  • 一次给十个需求 — 拆开，每次一件
  • 跑了一天没看过 — 长任务也要中途瞄两眼

🚨 一条铁律

它会犯错，你要兜底。它能跑命令、能改文件、能 git push,所以你必须在合并到 main / 推到测试环境之前看一眼 diff 和命令。AI 出错的成本 = 你能不能及时发现。

把它当一个聪明但缺乏项目经验的实习同事，你 review 的姿势就对了。
